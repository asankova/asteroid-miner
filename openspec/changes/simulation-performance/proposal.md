## Why

A game with 200+ robots, 2000+ asteroids, processing chains, particle effects, and AI agents will collapse under naive simulation. Performance must be architecturally designed, not bolted on. Critically, simulation performance is also a *game mechanic*: the player manages a "Compute Budget" resource that distributes processing fidelity across the field — near operations run at full tick resolution, distant ones at low resolution — which mirrors real distributed systems and creates interesting tradeoffs.

## What Changes

- Implement **spatial hash grid** for O(1) neighbor queries: all entities register position updates; collision, proximity, and signal detection queries cost constant time
- Implement **LOD simulation**: entities are bucketed by distance from camera into 3 simulation tiers:
  - **High fidelity**: full behavior tree tick every frame (200m radius around camera)
  - **Medium fidelity**: behavior tree ticks at 1/4 rate; positions interpolated (200–1000m)
  - **Low fidelity**: aggregate state update only (position delta, inventory delta) every 2s; individual robot behavior suppressed (>1000m)
- Implement **Web Workers** for heavy computation: behavior tree evaluation, yield calculation, and research RP accumulation run in worker threads off the main thread
- Introduce **Compute Budget** as a player-visible resource: the player allocates Compute Points across sectors; sectors with more points run at higher simulation fidelity; Compute Points are generated by Science Labs and upgraded Relay Beacons
- Compute Budget creates emergent gameplay: underfunded sectors degrade silently; attacks from ancient-race entities in low-compute sectors are more dangerous because reaction time is slower
- Add simulation performance overlay: F3-style debug panel showing per-sector tick rate, entity count, and compute allocation

## Capabilities

### New Capabilities
- `spatial-hash-grid`: O(1) spatial lookup structure updated each simulation tick
- `lod-simulation`: Distance-based simulation fidelity bucketing with behavior tree tick throttling
- `worker-thread-pool`: Web Worker pool for off-main-thread computation dispatch
- `compute-budget`: Player-managed resource allocating simulation fidelity across sectors
- `perf-overlay`: Debug/gameplay panel showing simulation stats per sector

### Modified Capabilities
- `behavior-tree`: Gains tick-rate multiplier from LOD-simulation fidelity bucket (from robot-agent-system)
- `robot-comms`: Message delivery rate degrades gracefully in low-fidelity sectors (from robot-agent-system)

## Impact

- Must be implemented before large-scale testing of any other system
- Compute Budget as a resource creates a unique design constraint: the game is explicitly about managing intelligence at scale
- Worker thread architecture means behavior tree evaluation is async — all dependent systems must handle async results
- Compute Budget ties directly to the LLM strategy agent layer: high-tier agents consume more Compute Points
