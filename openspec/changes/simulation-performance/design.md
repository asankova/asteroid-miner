## Context

The simulation must handle 200+ robots, 2000+ asteroids, 20+ processing structures, AI agent reasoning, and narrative state evaluation — all without blocking the main thread. The key architectural insight is that simulation fidelity is itself a resource the player manages. Entities near the camera run at full fidelity; entities in distant sectors run at aggregate fidelity. This is not just a performance hack — it is an explicit game mechanic via the Compute Budget resource.

## Goals / Non-Goals

**Goals:**
- Spatial hash grid for O(1) proximity queries
- Three-tier LOD simulation with per-entity fidelity bucketing
- Web Worker pool offloading heavy computation from main thread
- Compute Budget as a player-visible resource and gameplay mechanic
- Performance overlay for debugging and player-facing sector stats

**Non-Goals:**
- GPU-based simulation (simulation runs on CPU in workers; GPU is rendering only)
- Deterministic lockstep simulation (single-player; no network sync requirement)
- Sub-millisecond precision timing (10ms tick granularity is sufficient)

## Decisions

**Spatial hash with cell size = 200 world units**
Cell size chosen so average entities per cell is 2–5 at max field density. Query radius for behavior tree neighbor checks is typically 300–500 units, requiring at most 4 cells to scan. Alternative: k-d tree — better for static data but has high rebuild cost for moving robots.

**Three fidelity tiers: HIGH (full tick), MEDIUM (1/4 tick), LOW (aggregate only)**
High: within 200m of camera. Medium: 200–1000m. Low: >1000m. Low-fidelity robots don't run behavior trees; their aggregate state (total ore collected, total distance moved) is updated from a simplified model. This is the "simulation depth" concept: players see that far-away operations are running but not in full resolution.

**Worker pool: 3 workers — simulation, physics/mining, research/metrics**
Separating concerns by worker prevents one slow computation from stalling all simulation. Workers communicate via postMessage with structured clone. SharedArrayBuffer used only for telemetry ring buffer where latency matters.

**Compute Budget generated by Science Labs and upgraded Relay Beacons**
Compute Points (CP) are a soft resource: the simulation always runs, but sector fidelity upgrades require CP. Default: camera-adjacent sector always HIGH regardless of CP. Player allocates CP to other sectors to raise them from LOW to MEDIUM or HIGH. CP connects gameplay progression (build better infrastructure) to simulation quality (watch more of your operation in real resolution).

## Risks / Trade-offs

[Worker postMessage serialization cost for large state updates] → Use transferable ArrayBuffers for bulk data; postMessage with structured clone for small messages.

[LOD tier transitions (robot moving between fidelity zones) require state handoff] → On tier transition, serialize robot state from source tier and deserialize in destination tier. Small state object (~200 bytes); cost is negligible.

[Compute Budget may confuse players used to simulation games "just working"] → Add tutorial tooltip explaining the mechanic; frame it as "simulation resolution" rather than "performance."
